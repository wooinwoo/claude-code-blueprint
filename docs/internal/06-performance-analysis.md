## 1. 토큰 소모 구조

### 매 세션 고정 비용 (항상 로드되는 것)

Claude Code 세션이 시작되면 아래가 **무조건** 컨텍스트에 들어갑니다:

| 항목 | 추정 토큰 | 설명 |
|------|-----------|------|
| CLAUDE.md | 200~800 | 프로젝트 설명 (200줄 이하 권장) |
| Rules (paths 없는 것) | 1,000~4,000 | 매 세션 전문 로드 |
| Skills descriptions | 500~1,500 | description만 (1-2줄 × 스킬 수) |
| Auto Memory (MEMORY.md) | 200~800 | 처음 200줄 |
| **총 고정 비용** | **~2,000~7,000** | |

### 프로필별 고정 비용 비교

| 프로필 | Rules 수 | Rules 추정 토큰 | Skills description | 총 고정 비용 |
|--------|---------|----------------|-------------------|-------------|
| react-next | 21개 | ~4,000 | ~800 (19개) | ~5,800 |
| nestjs | 18개 | ~3,500 | ~700 (17개) | ~5,200 |
| fullstack | 23개 | ~4,500 | ~800 (20개) | ~6,300 |
| designer | 8개 | ~2,000 | ~350 (7개) | ~3,350 |
| planner | 7개 | ~1,800 | ~250 (5개) | ~3,050 |

**fullstack이 가장 비쌈.** react + nestjs 룰을 모두 갖고 있어서.

**planner가 가장 저렴.** 개발 룰이 없고 스킬도 적어서.

### 비용 절약 방법

1. **Rules에 `paths` 추가**: 특정 파일 작업 시만 로드되게 변경
   - react-composition.md에 `paths: ["src/components/**/*.tsx"]` 넣으면 컴포넌트 작업 시만 로드
   - 절약 효과: 룰당 200~500 토큰

2. **긴 규칙은 Skills로 이동**: Rules에서 Skills로 옮기면 description(~50 토큰)만 상시 로드
   - 200줄 룰 → Skills로 이동하면 ~800 토큰 절약

3. **불필요한 base 룰 제거**: `paths`가 없는 base-common 룰 9개가 매 세션 로드
   - non-dev는 이미 2개로 필터링됨
   - dev도 필요 없는 룰이 있으면 setup.ps1에서 필터링 추가 가능

## 3. 작업 시간 추정

### /orchestrate 총 소요 시간

| Phase | Standard | Full |
|-------|----------|------|
| Phase 0: Scan | - | 2-5분 |
| Phase 1: Plan + 승인 | 3-5분 | 5-10분 |
| Phase 2: Branch | 1분 | 1분 |
| Phase 3: Develop | 10-30분 | 20-60분 |
| Phase 4: Review + PR | 3-10분 | 10-20분 |
| **총** | **17-46분** | **38-96분** |

Phase 3이 가장 오래 걸림. 기능 규모에 따라 편차 큼.

### 병목 지점

1. **Phase 3 검증 루프**: lint→type→build→test 실패 시 최대 3회 반복. 대형 프로젝트면 빌드만 2-3분
2. **Phase 4 에이전트 리뷰**: Full 모드에서 8개 에이전트가 순차 실행. 병렬이 아님
3. **pnpm install**: 워크트리에서 처음 실행하면 1-3분
4. **Lighthouse**: 페이지당 15-30초. 5페이지면 2분

### 시간 절약 방법

1. **Standard 모드를 기본으로**: Full은 중요한 기능에만. 일상 작업은 Standard
2. **Phase 0 스킵**: Standard는 자동 스킵. 시간 2-5분 절약
3. **에이전트 선별 투입**: 변경 파일에 따라 불필요한 에이전트 자동 스킵 (이미 구현됨)
4. **캐시 활용**: pnpm store가 있으면 install이 빠름

## 4. 결과물 품질 비교

### Standard vs Full

| 항목 | Standard | Full |
|------|----------|------|
| 코드 품질 | 중상 | 상 |
| 테스트 커버리지 | 낮음 (테스트 선택적) | 높음 (TDD 강제) |
| 설계 품질 | 간단한 플랜 | Opus architect 심층 설계 |
| 리뷰 깊이 | 필수 에이전트만 (2-4개) | 전원 투입 2라운드 (6-8개) |
| 토큰 비용 | 1x | 2-3x |
| 소요 시간 | 1x | 2-3x |

### 언제 Full을 쓰나

- 신규 기능 (기존 코드 참조 필요) → Phase 0 코드베이스 스캔이 유용
- 복잡한 아키텍처 변경 → Opus architect 설계가 유용
- 프로덕션 배포 전 마지막 기능 → 전원 리뷰가 유용
- TDD가 필요한 핵심 로직 → TDD 모드 강제

### 언제 Standard로 충분한가

- UI 수정, 스타일 변경
- 기존 패턴 따르는 CRUD 추가
- 버그 수정 (이건 `/fix`가 더 적합)
- 작은 기능 추가 (파일 1-5개)

## 5. 비용 최적화 권장사항

### 즉시 적용 가능

| 방법 | 절약 | 부작용 |
|------|------|--------|
| Rules에 `paths` 추가 | 세션당 500~2,000 토큰 | 없음 |
| 불필요한 base 룰 제거 | 세션당 200~800 토큰 | 해당 규칙 미적용 |
| Standard 기본 사용 | orchestrate당 50~100K 토큰 | 리뷰 깊이 감소 |
| 에이전트 model 조정 | 에이전트당 30-50% 절약 | 분석 품질 감소 |

### 에이전트 모델 최적화

현재 대부분 sonnet. 탐색/분석 위주 에이전트는 haiku로 바꿔도 됨:

| 에이전트 | 현재 | 권장 | 이유 |
|---------|------|------|------|
| convention-reviewer | sonnet | haiku | 패턴 매칭 위주, 깊은 분석 불필요 |
| impact-analyzer | sonnet | haiku | 파일 탐색 위주 |
| doc-updater | haiku | haiku | 이미 최적 |
| 나머지 | sonnet | sonnet | 유지 (깊은 분석 필요) |
| architect | opus | opus | 유지 (설계 품질 중요) |
| researcher-strategist | opus | opus | 유지 (전략 분석 품질 중요) |

### 장기 개선

| 방법 | 설명 |
|------|------|
| 토큰 대시보드 | 세션별/커맨드별 토큰 사용량 추적 |
| 자동 모드 추천 | 변경 규모에 따라 Standard/Full 자동 선택 |
| 에이전트 병렬화 | Phase 4 리뷰를 순차→병렬로 (시간 절약) |
| 증분 스캔 | Phase 0에서 이전 스캔 결과 캐시, 변경분만 재스캔 |
