# PM 문서 품질 게이트

plans/ 하위 문서 작성 시 Claude가 자주 놓치는 품질 기준을 검사하는 규칙.
문서 구조 자체는 각 커맨드(prd, roadmap 등)가 정의하므로, 여기서는 **공통 품질 기준만** 다룬다.

---

## 1. 파일명 규칙 [CRITICAL]

**kebab-case + 날짜** 필수. 공백, 대문자, 한글 파일명 금지.

```
GOOD: prd-user-auth-2026-03-14.md
GOOD: research-ai-market-2026-03.md
GOOD: retro-sprint-12-2026-03-14.md

BAD:  PRD.md              ← 날짜 없음, 대문자
BAD:  my document.md      ← 공백
BAD:  사용자인증PRD.md     ← 한글 + 대문자
BAD:  prd_user_auth.md    ← snake_case
```

파일명 패턴: `{문서타입}-{주제}-{YYYY-MM-DD}.md`

---

## 2. 필수 메타데이터 [CRITICAL]

모든 plans/ 문서 상단에 아래 4개 필드 필수. 하나라도 누락되면 Draft 상태로 간주.

```markdown
- **작성자**: 김PM
- **작성일**: 2026-03-14
- **상태**: Draft / Review / Approved / Archived
- **관련**: PROJ-1234 (Jira 키 또는 OKR/에픽 링크)
```

Claude가 자주 틀리는 패턴:
- 작성일을 "2026년 3월"처럼 모호하게 → **반드시 YYYY-MM-DD**
- 상태를 빠뜨림 → **최소 Draft라도 명시**
- 관련 링크 없이 "추후 연결" → **Jira 키가 없으면 "없음 — 생성 필요" 표기**

---

## 3. TBD 규칙 [HIGH]

TBD 사용 가능. 단, **해결 기한 + 담당자** 반드시 명시.

```
BAD:  TBD
BAD:  추후 결정
BAD:  TBD (나중에 논의)

GOOD: TBD (3/20까지 결정 — 담당: 김PM)
GOOD: TBD (디자인 확정 후 반영 — 담당: 이디자이너, 기한: 3/25)
```

문서 내 TBD가 3개 이상이면 문서 상단에 **TBD 요약 테이블** 추가:

```markdown
## 미결 사항 (TBD)
| # | 항목 | 담당 | 기한 | 비고 |
|---|------|------|------|------|
| 1 | 결제 수단 범위 | 김PM | 3/20 | PG사 미팅 후 결정 |
| 2 | 알림 채널 | 박기획 | 3/22 | 사용자 조사 진행 중 |
```

---

## 4. 수치 필수 — 정성적 표현 금지 [HIGH]

"많다/적다/빠르다/느리다/크다/작다" 등 모호한 표현 금지.
모든 주장에 **구체적 수치 + 출처 + 기준 시점** 포함.

```
BAD:  "사용자가 많이 이탈한다"
GOOD: "퍼널 3단계에서 34% 이탈 (GA 기준, 2026-03 데이터)"

BAD:  "로딩이 느리다"
GOOD: "메인 페이지 LCP 4.2초 (Lighthouse, 2026-03-10 측정)"

BAD:  "시장이 빠르게 성장 중"
GOOD: "CAGR 23.4% (2024-2028, Gartner 2025)"

BAD:  "대부분의 사용자가 선호"
GOOD: "설문 응답자 78% 선호 (n=342, 2026-02 사내 조사)"
```

수치를 모르면 "데이터 필요 — 담당: OOO, 기한: OO/OO"로 표기. 추측 수치 사용 금지.

---

## 5. 출처 필수 [HIGH]

외부 데이터 인용 시 **URL 또는 소스명 + 발행일** 필수.

```
BAD:  "시장 규모는 약 100억이다"
GOOD: "시장 규모 $10B (Gartner, 2025-06)"
GOOD: "MAU 1.2억 (출처: https://example.com/report, 2025-12)"
```

자체 데이터는 **측정 도구 + 기간** 명시:
```
GOOD: "전환율 3.2% (GA4, 2026-01~02 평균)"
GOOD: "CS 인입 847건/월 (Zendesk, 2026-02)"
```

---

## 6. 액션 아이템 형식 [HIGH]

**"누가 + 무엇을 + 언제까지"** 3요소 필수.
문서 끝 Next Steps에 테이블로 정리.

```
BAD:  "추후 논의 필요"
BAD:  "검토 후 진행"
BAD:  "디자인팀에서 처리"     ← 누가? 언제까지?

GOOD:
| 항목 | 담당 | 기한 |
|------|------|------|
| API 스펙 확정 | 김PM | 3/18 |
| 디자인 시안 v1 | 이디자이너 | 3/20 |
| 기술 검토 미팅 | 박리드 | 3/19 |
```

기한이 정해지지 않았으면 "OO 이후 1주 내"처럼 상대 기한이라도 명시.

## 7. 문단 및 구조 규칙 [MEDIUM]

### 문단 길이
- 최대 **4줄**. 초과 시 분리하거나 bullet point로 전환.
- Claude는 설명을 장문으로 쓰는 경향 → 항상 4줄 체크.

### Heading 계층
- h1: 문서 제목 (1개만)
- h2: 섹션
- h3: 서브섹션
- **h4 이상 금지** — 깊이가 필요하면 구조를 재설계하거나 별도 문서로 분리.

### 테이블 활용
- **3개 이상 비교/나열은 반드시 테이블** 사용.
- 산문으로 "A는 이렇고, B는 저렇고, C는 또 다르다"식 나열 금지.

```
BAD:
Slack은 월 $7.25이고 Teams는 무료이며 Discord는 $9.99입니다.

GOOD:
| 도구 | 월 비용 | 특징 |
|------|---------|------|
| Slack | $7.25 | 서드파티 연동 풍부 |
| Teams | 무료 (M365) | MS 생태계 통합 |
| Discord | $9.99 | 커뮤니티 중심 |
```

## 8. 검수 체크리스트 [MEDIUM]

문서 작성 완료 후 아래 항목 자가 검수:

- [ ] 파일명이 kebab-case + 날짜인가?
- [ ] 메타데이터 4개 필드(작성자/작성일/상태/관련) 모두 있는가?
- [ ] TBD에 기한과 담당자가 있는가?
- [ ] "많다/적다/빠르다" 같은 모호한 표현이 없는가?
- [ ] 모든 외부 데이터에 출처가 있는가?
- [ ] 액션 아이템이 "누가+무엇을+언제까지" 형식인가?
- [ ] 4줄 넘는 문단이 없는가?
- [ ] h4 이상 heading이 없는가?
- [ ] 3개 이상 나열을 산문으로 쓴 곳이 없는가?
