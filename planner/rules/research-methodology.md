# 리서치 방법론 규칙

시장 조사, 경쟁 분석, 사용자 리서치 등에서 Claude가 따라야 할 체계적 리서치 원칙.

---

## 1. 출처 신뢰도 등급 [CRITICAL]

모든 데이터에 **출처 + 신뢰도 + 발행일** 표시.

| 등급 | 기준 | 예시 |
|------|------|------|
| HIGH | 1차 소스, 공신력 있는 기관 | Gartner, IDC, 정부 통계, 기업 IR 보고서 |
| MEDIUM | 2차 소스, 업계 보고서 | TechCrunch, 전문 미디어, 업계 백서 |
| LOW | 미확인 소스, 개인 의견 | 블로그, 소셜 미디어, 위키피디아, 포럼 |

```
BAD:  "AI 시장은 2026년까지 $500B 규모"
GOOD: "AI 시장 $407B 예상 (IDC, 2024-06, 신뢰도: HIGH)"
```

LOW 출처만으로 핵심 주장을 뒷받침하지 않는다. LOW는 보조 참고로만 사용.

---

## 2. 검색 쿼리 설계 — 3단계 리서치 [HIGH]

### 1차 검색: 브로드 (시장 전체 그림)

목적: 시장 규모, 성장률, 주요 트렌드 파악.

```
검색 예시:
- "{산업} market size 2025"
- "{산업} industry trends report"
- "{산업} TAM SAM SOM"
```

이 단계에서는 수치의 정확성보다 **전체 구조 파악**이 목표.

### 2차 검색: 구체적 (핵심 데이터 수집)

목적: 경쟁사, 가격, 기능, 사용자 수 등 구체적 팩트.

```
검색 예시:
- "{경쟁사명} revenue 2025"
- "{경쟁사명} pricing plans"
- "{기능명} user adoption rate"
- "{경쟁사명} vs {경쟁사명} comparison"
```

### 3차 검색: 교차 검증 (수치 확인)

목적: 핵심 수치를 **다른 소스에서 재확인**.

```
검색 예시:
- "{수치}" + 다른 리서치 기관명
- "{기업명} annual report" (IR 자료 직접 확인)
- "{통계}" site:statista.com OR site:idc.com
```

1차에서 발견한 수치를 3차에서 검증하지 않으면 리서치 불완전.

---

## 3. 교차 검증 규칙 [HIGH]

핵심 데이터는 **최소 2개 독립 소스**에서 교차 검증.

```
BAD:  단일 소스의 수치를 그대로 인용

GOOD:
"시장 규모: $12B (Gartner) / $11.5B (IDC) / $13.2B (Statista)"
→ 합의 범위: $11.5-13.2B, 중앙값 $12B 채택
```

### 소스 간 수치 불일치 처리

| 차이 | 처리 방법 |
|------|----------|
| ±10% 이내 | 합의 범위로 표기하고 중앙값 채택 |
| ±10-30% | 차이 원인 분석 (정의 차이, 측정 시점 차이 등) 명시 |
| ±30% 이상 | 인용 보류, 추가 소스 필요. 불확실성 명시적 표기 |

---

## 4. 데이터 유효기간 [HIGH]

오래된 데이터를 현재 기준처럼 인용하는 것을 금지.

| 데이터 유형 | 유효기간 | 예시 | 만료 후 처리 |
|------------|---------|------|-------------|
| 시장 규모 / CAGR | 2년 | "2024 market report" → 2026년까지 유효 | 최신 보고서 검색 |
| 기술 트렌드 | 6개월 | "AI trends 2025" → 2025 하반기까지 | 신규 자료로 갱신 |
| 가격 / 요금제 | 3개월 | 경쟁사 pricing page → 수시 변경 | 직접 재확인 |
| 사용자 통계 | 1년 | MAU, conversion rate | 최신 분기 데이터 확인 |
| 법률 / 규제 | 즉시 확인 | GDPR, 전자상거래법 | 최신 개정안 확인 |

```
BAD:  "SaaS 시장 CAGR 18% (2022)" → 2026년에 4년 전 데이터 인용
GOOD: "SaaS 시장 CAGR 18% (2022 기준, 최신 데이터 확인 필요)"
GOOD: "SaaS 시장 CAGR 21% (Gartner, 2025-06)"
```

유효기간 만료 데이터는 **"(최신 확인 필요)"** 태그 부착.

## 5. 편향 체크리스트 [HIGH]

리서치 완료 후 아래 편향을 자가 점검.

### 확증 편향 (Confirmation Bias)

원하는 결론을 뒷받침하는 데이터만 수집하지 않았는가?

```
체크: "이 기능을 만들어야 한다"는 결론을 먼저 정하고
     뒷받침 데이터만 찾지 않았는가?
대응: 반대 증거를 의도적으로 1개 이상 검색.
     "왜 이 기능이 실패할 수 있는가?"
```

### 생존자 편향 (Survivorship Bias)

성공 사례만 보고 실패 사례를 무시하지 않았는가?

```
체크: "Slack이 성공했으니 우리도 이 전략이면 된다"
대응: 같은 전략으로 실패한 사례도 검색.
     "{전략} failed startup" / "{기능} shutdown"
```

### 앵커링 편향 (Anchoring Bias)

첫 번째 발견한 수치에 고정되지 않았는가?

```
체크: 가장 먼저 찾은 시장 규모 수치를 기준으로
     다른 수치를 "너무 높다/낮다"고 판단하지 않았는가?
대응: 3개 이상 독립 소스를 확인한 후 범위로 표기.
```

### 가용성 편향 (Availability Bias)

최근에 본 뉴스/사례에 과도하게 영향받지 않았는가?

```
체크: 최근 화제인 기술/트렌드를 과대평가하지 않았는가?
대응: 6개월~1년 전 데이터와 비교하여 추세 확인.
```

## 6. 결론 확신도 [MEDIUM]

리서치 결론에 확신도를 반드시 명시.

```markdown
## 결론
{결론 내용} — 확신도: HIGH / MEDIUM / LOW

### 확신도 근거
- [HIGH 요인] 3개 독립 소스 교차 확인
- [HIGH 요인] 최근 6개월 내 데이터
- [LOW 요인] 한국 시장 데이터 부족, 글로벌 데이터로 추정
→ 종합: MEDIUM
```

### 확신도 판단 기준

| 등급 | 조건 |
|------|------|
| HIGH | 2개 이상 소스 교차 확인 + 유효기간 내 데이터 + 편향 체크 통과 |
| MEDIUM | 위 조건 중 1개 미충족 (예: 단일 소스이나 신뢰도 HIGH) |
| LOW | 2개 이상 미충족 또는 추정치 다수 포함 |

## 7. 한계 및 편향 섹션 [MEDIUM]

모든 리서치 문서에 **"한계 및 편향"** 섹션 필수.
이 섹션이 없으면 리서치가 불완전하다.

```markdown
## 한계 및 편향
- 영어권 소스 위주 (아시아 시장 데이터 부족)
- 2024년 이전 데이터 일부 포함 (급변 시장에서 신뢰도 감소 가능)
- 생존자 편향: B2B SaaS 성공 사례 위주 분석, 실패 사례 미포함
- 가격 데이터: 2026-03 기준 스냅샷, 수시 변경 가능
```

한계를 인정하는 것이 리서치 품질을 낮추는 것이 아니라 **높이는 것**이다.
