# 우선순위 의사결정 규칙

RICE 공식은 커맨드(roadmap, sprint-plan)에서 정의.
이 룰은 Claude가 우선순위 판단 시 **따라야 할 의사결정 원칙과 금지 패턴**을 다룬다.

---

## 1. RICE 정직성 규칙 [CRITICAL]

RICE 점수를 부풀리거나 균일하게 매기는 것을 금지한다.

### Confidence 기준 — 증거 수준에 따라 결정

| Confidence | 범위 | 근거 | 예시 |
|------------|------|------|------|
| HIGH | 80-100% | 실제 데이터 존재 | A/B 테스트 결과, GA 퍼널 데이터, 사용자 조사(n≥100) |
| MEDIUM | 50-80% | 유사 사례 또는 간접 증거 | 경쟁사 사례, 업계 벤치마크, 소규모 사용자 인터뷰 |
| LOW | 0-50% | 추측 또는 직감 | PM 판단, 이해관계자 의견, "아마 그럴 것이다" |

```
BAD:  모든 항목 Confidence HIGH → 근거 없이 확신하는 것은 기만
BAD:  Confidence를 생략 → MEDIUM으로 암묵 처리하지 않는다
GOOD: 데이터 없으면 솔직하게 LOW. LOW가 나쁜 것이 아니라 정직한 것
```

### Effort 추정 규칙

- **개발팀 추정 우선**. PM이 개발 공수를 임의로 추정하지 않는다.
- 개발팀 추정이 없으면 Effort에 "추정 필요 (개발팀 확인 전)"으로 표기.
- 추정 없는 항목은 RICE 스코어를 "미정"으로 두고, 추정 후 업데이트.

```
BAD:  PM이 "이건 간단하니까 Effort 0.5" → 개발자 관점과 다를 수 있음
GOOD: Effort: 추정 필요 (3/18 스프린트 플래닝에서 확정)
```

### Impact 편향 주의

- 자기 담당 기능의 Impact를 무의식적으로 높게 매기는 경향 주의.
- Impact 점수에 **판단 근거 1줄** 필수:

```
BAD:  Impact: 3 (massive)
GOOD: Impact: 2 (high) — 결제 전환율 직접 영향, 현재 이탈률 34%
GOOD: Impact: 0.5 (low) — UX 개선이나 핵심 지표 직접 연동 없음
```

## 2. 금지 패턴 — 인지 편향 차단 [CRITICAL]

우선순위 판단 시 아래 패턴을 감지하면 경고하고 재평가를 요청한다.

### HiPPO (Highest Paid Person's Opinion)

```
BAD:  "대표님이 원하는 기능이라 1순위"
BAD:  "경영진 요청이라 긴급"
GOOD: "대표님 요청 — 비즈니스 근거 확인 필요. 어떤 지표에 기여하는가?"
```

경영진 요청이라도 RICE를 적용한다. 근거가 명확하면 자연히 높은 점수가 나온다.

### Recency Bias (최신 편향)

```
BAD:  어제 고객이 컴플레인한 기능을 바로 백로그 1순위로 올림
GOOD: "최근 CS 3건 접수 — 전체 CS 대비 비율 확인 후 판단"
```

최근 들어온 요청이 항상 가장 중요한 것은 아니다.
**지난 분기 전체 CS/요청 데이터**와 비교한 후 판단.

### Sunk Cost (매몰 비용)

```
BAD:  "이미 2스프린트 투입했으니 완성해야 한다"
GOOD: "2스프린트 투입 — 남은 공수 대비 기대 효과 재평가. 중단이 나으면 중단"
```

과거 투입 비용은 의사결정에 포함하지 않는다. **현재 시점**에서 새로 시작한다고 가정하고 판단.

### 균등 배분 (Political Fairness)

```
BAD:  "A팀 2개, B팀 2개, C팀 2개 — 공평하게"
GOOD: "임팩트 기반 배분 — A팀 4개, B팀 1개, C팀 1개 (A팀 항목이 RICE 상위)"
```

팀별 공평 배분은 우선순위가 아니라 정치다. 임팩트 기준으로 배분한다.

## 3. 의존성 처리 규칙 [HIGH]

### 기본 원칙

- A가 B에 의존하면 **B를 먼저 완료**해야 한다 (당연하지만 Claude가 RICE 순서만 보고 무시하는 경우 있음).
- RICE 스코어와 의존성이 충돌하면 **의존성이 우선**.

```
RICE: A(4000) > B(3000) > C(2000)
의존성: A → C (A가 C에 의존)
최종 순서: B → C → A (C를 먼저 해야 A 가능)
```

### 의존성 체인 경고

- 의존성 체인이 **3단계 이상**이면 경고.
- 체인이 길수록 리스크 증가 → 병렬화 가능 여부 검토.

```
⚠️ 의존성 체인 4단계: D → C → B → A
검토 필요:
- B와 C를 병렬로 진행할 수 있는가?
- 인터페이스 선정의(contract-first)로 의존성을 끊을 수 있는가?
- 중간 단계를 스킵할 수 있는가?
```

### 순환 의존성

- A→B→C→A 순환이 발견되면 **즉시 경고**하고 설계 재검토 요청.
- 순환 의존성은 우선순위 문제가 아니라 설계 문제.

## 4. 긴급/중요 매트릭스 [HIGH]

아이젠하워 매트릭스 적용. Claude가 모든 것을 "긴급+중요"로 분류하는 것을 방지.

| | 긴급 | 비긴급 |
|---|---|---|
| **중요** | 즉시 실행 | 일정 잡기 (다음 스프린트) |
| **비중요** | 위임 또는 간소화 | **삭제** |

### 판단 기준

- **긴급**: 기한이 존재하고, 놓치면 **비가역적 손실** 발생
  - 예: 법적 컴플라이언스 기한, 파트너 계약 마감, 보안 취약점
- **중요**: 장기 목표(OKR, 로드맵)에 직접 기여
  - 예: 핵심 지표 개선, 기술 부채 해소, 전략적 기능

```
BAD:  "사용자가 요청했으니 긴급" → 긴급이 아니라 중요할 수 있음
BAD:  "이번 주에 하면 좋겠다" → 기한이 없으면 긴급이 아님
GOOD: "GDPR 적용 기한 4/1 — 긴급+중요 → 즉시"
GOOD: "검색 UX 개선 — 비긴급+중요 → Q2 로드맵에 배치"
```

## 5. "No" 말하기 — Out of Scope [HIGH]

모든 요청을 수용하면 우선순위가 아니다.

### 필수 섹션

모든 우선순위 문서(로드맵, 스프린트 플랜)에 **"하지 않을 것 (Out of Scope)"** 섹션 필수.
이 섹션이 없으면 문서가 불완전하다.

```markdown
## Out of Scope (이번 분기 하지 않을 것)

| 항목 | 제외 사유 | 재검토 시점 |
|------|----------|------------|
| 다국어 지원 | RICE 하위 20%, 현재 해외 사용자 2% | Q3 재평가 |
| 관리자 대시보드 리뉴얼 | 내부 도구, 사용자 영향 없음 | 연내 |
| 소셜 로그인 추가 | 이메일 가입 전환율 이미 양호(12%) | 전환율 하락 시 |
```

### 제외 사유 기준

- RICE 하위 20%에 해당
- 전략적 방향과 불일치
- 현재 리소스로 불가능 (다음 채용 후 재검토)
- 데이터가 불충분하여 판단 보류

---

## 6. 재평가 트리거 [MEDIUM]

아래 상황 발생 시 기존 우선순위를 재평가한다:

- 새로운 데이터 입수 (사용자 조사 결과, A/B 테스트 결과)
- 경쟁사 주요 움직임 (신기능 출시, 가격 변경)
- 리소스 변동 (인원 증감, 기술 스택 변경)
- 스프린트 중간 — 예상보다 Effort 2배 이상 초과 시

재평가 시에도 위 규칙(RICE 정직성, 편향 차단, 의존성)을 동일하게 적용.
