# Stakeholder Communication Skill

이해관계자 커뮤니케이션 — 보고, 상태 업데이트, 의사결정 요청, 나쁜 소식 전달.

## When to Activate

- 경영진/리더십 보고 작성
- 주간/월간 상태 업데이트
- 의사결정 요청 문서
- 크로스팀 협업 요청
- 나쁜 소식(지연, 장애, 비용 초과) 전달

## Audience Adaptation Matrix

| 청중 | 관심사 | 적정 길이 | 용어 수준 | 시작 패턴 |
|------|--------|-----------|-----------|-----------|
| C-Level (CEO/CTO) | ROI, 전략적 임팩트, 리스크 | 1페이지 | 비즈니스 용어 | "이번 분기 매출에 X% 영향..." |
| VP/Director | 로드맵 영향, 리소스, 일정 | 2-3페이지 | 도메인 용어 OK | "현재 진행률 X%, 리스크 Y건..." |
| 개발팀 | 기술 스펙, API, 제약사항 | 상세 | 기술 용어 OK | "입력: X, 출력: Y, 제약: Z" |
| 디자인팀 | 사용자 플로우, 인터랙션 | 중간 | UX 용어 OK | "사용자가 A에서 B로 이동 시..." |
| 외부 파트너 | SLA, 인터페이스, 일정 | 공식적 | 중립 용어 | 계약서 톤 |

**적용 규칙**: 청중이 혼합이면 가장 높은 직급 기준으로 길이/용어를 맞추고, 상세 내용은 부록으로 분리.

## Communication Templates

### Executive Summary Template

```markdown
**상황**: [현재 상태 1문장]
**임팩트**: [비즈니스 영향 수치]
**요청**: [의사결정 사항]
**옵션**:
| 옵션 | 장점 | 단점 | 비용 | 추천 |
|------|------|------|------|------|
| A | ... | ... | ... | ✅ |
| B | ... | ... | ... | |
**기한**: [언제까지 결정 필요]
```

### Status Update Template (Slack/이메일)

```markdown
🟢 정상 / 🟡 주의 / 🔴 위험

**이번 주**: [완료 항목 2-3개]
**다음 주**: [계획 2-3개]
**블로커**: [있으면 1-2개, 없으면 "없음"]
```

### Bad News Delivery Template

나쁜 소식은 다음 순서를 엄격히 지킨다:

```markdown
1. 사실 먼저: "X 기능 출시가 2주 지연됩니다"
2. 원인: "외부 API 연동 이슈로..."
3. 영향: "매출 영향 약 X%..."
4. 대응: "A안/B안을 검토 중이며..."
5. 요청: "Y까지 A/B 중 결정 부탁드립니다"
```

**금지**: "좋은 소식부터 말씀드리면..." 으로 시작하는 good news sandwich. 나쁜 소식이 묻힌다.

### Decision Request Template

```markdown
## 의사결정 요청: [주제]

**배경**: [1-2문장 — 왜 이 결정이 필요한가]
**옵션**:
| 옵션 | 장점 | 단점 | 비용 | 일정 영향 |
|------|------|------|------|-----------|
| A    |      |      |      |           |
| B    |      |      |      |           |

**추천**: 옵션 [X] — [1문장 이유]
**기한**: [결정 필요 시점] — [늦으면 어떤 영향이 있는지]
```

## Writing Rules

### 핵심 원칙

1. **결론 먼저 (Pyramid Principle)**: 배경 → 결론이 아니라, 결론 → 근거 순서. 첫 문장이 가장 중요한 메시지여야 한다.
2. **숫자로 말하기**: "많이 개선됨" → "전환율 12%→18% (50% 개선)". 정량화할 수 없으면 최소한 규모감을 준다 ("약 200건", "3개 팀 영향").
3. **나쁜 소식 먼저**: good news sandwich 금지. 리스크/지연을 숨기면 신뢰를 잃는다.
4. **액션 아이템 = 누가 + 무엇을 + 언제까지**: "논의 필요" → "@김PM이 API 방식 A/B 중 3/15까지 결정"
5. **한 문서 한 요청**: 여러 요청을 섞으면 아무것도 결정되지 않는다.

### 이메일/Slack 제목 규칙

형식: `[프로젝트명] 상태/요청/공유 — 핵심 내용`

예시:
- `[결제v2] 요청 — API 연동 방식 결정 (3/15까지)`
- `[검색개선] 공유 — A/B 테스트 결과 (+12% CTR)`
- `[인프라] 주의 — Redis 메모리 85% 도달, 증설 필요`

**분류 키워드**:
- `요청`: 상대방의 액션이 필요
- `공유`: 참고용, 액션 불필요
- `주의`: 리스크 알림
- `긴급`: 즉시 대응 필요 (남발 금지)

### 길이 가이드

| 채널 | 최대 길이 | 초과 시 |
|------|-----------|---------|
| Slack 메시지 | 5줄 | 스레드 또는 문서 링크 |
| 이메일 | 스크롤 1회 | 요약 + 첨부 |
| 보고서 | 3페이지 | Executive Summary 1페이지 + 부록 |
| 회의 발언 | 2분 | 사전 문서 공유 |
