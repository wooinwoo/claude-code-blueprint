# Stakeholder Communication Skill

이해관계자 커뮤니케이션. 경영진 보고, 상태 업데이트, 의사결정 요청, 대상별 메시지 조정.

## When to Activate

- 경영진/리더십에 보고할 때
- 주간/월간 상태 업데이트 작성
- 의사결정 요청 문서 작성
- 크로스팀 협업 요청

## Audience Adaptation

| 대상 | 관심사 | 깊이 | 용어 수준 |
|------|--------|------|-----------|
| C-Level | 비즈니스 임팩트, ROI, 리스크 | 요약 (1페이지) | 비즈니스 용어 |
| VP/Director | 전략 정렬, 리소스, 일정 | 중간 (2-3페이지) | 비즈니스 + 도메인 |
| 개발팀 | 기술 요구사항, 제약, 우선순위 | 상세 | 기술 용어 OK |
| 디자인팀 | UX 요구사항, 사용자 흐름, 제약 | 상세 | 디자인 용어 OK |

## Templates

### Executive Summary (1페이지)
```markdown
## [제목]

**한 줄 요약**: [핵심 메시지]

**현황**: [현재 상태 — 수치 포함]
**요청**: [의사결정/승인/리소스 필요사항]
**임팩트**: [비즈니스 영향 — 매출/사용자/비용]
**리스크**: [안 하면 어떻게 되는가]
**일정**: [주요 마일스톤]
```

### Decision Request
```markdown
## 의사결정 요청: [주제]

**배경**: [1-2문장]
**옵션**:
| 옵션 | 장점 | 단점 | 비용 | 일정 |
|------|------|------|------|------|
| A    |      |      |      |      |
| B    |      |      |      |      |

**추천**: 옵션 [X] — [이유]
**필요한 결정**: [구체적 질문]
**기한**: [결정 필요 시점과 이유]
```

### Status Update (주간)
```markdown
## 주간 업데이트 — [날짜]

### TL;DR (3줄)
1. [가장 중요한 진행]
2. [주의 필요 사항]
3. [다음 주 핵심]

### 상세
| 항목 | 상태 | 메모 |
|------|------|------|

### 블로커/리스크
| 이슈 | 영향 | 요청 |
|------|------|------|
```

## Communication Principles

1. **결론 먼저**: 배경 설명 → 결론 순서가 아니라, 결론 → 근거 순서
2. **숫자로 말하기**: "많이 개선됨" → "전환율 12% → 18% (50% 개선)"
3. **액션 명확히**: "논의 필요" → "3/15까지 옵션 A/B 중 결정 필요"
4. **나쁜 소식 먼저**: 리스크/지연을 숨기지 않고 조기에 공유 + 대안 포함
